# 渗透测试全流程详解

## 一、PTES标准方法论概述

### 1.1 什么是PTES

PTES（Penetration Testing Execution Standard，渗透测试执行标准）是由安全社区共同制定的一套渗透测试标准框架，旨在为渗透测试提供统一的方法论和执行标准。PTES定义了渗透测试的七个阶段，为安全专业人员提供了系统化的测试流程。

### 1.2 PTES七大阶段

```
┌─────────────────────────────────────────────────────────────┐
│                    PTES渗透测试流程                          │
├─────────────────────────────────────────────────────────────┤
│  阶段1: 前期交互（Pre-Engagement Interactions）              │
│  阶段2: 信息收集（Intelligence Gathering）                   │
│  阶段3: 威胁建模（Threat Modeling）                          │
│  阶段4: 漏洞分析（Vulnerability Analysis）                   │
│  阶段5: 漏洞利用（Exploitation）                             │
│  阶段6: 后渗透（Post Exploitation）                          │
│  阶段7: 报告编写（Reporting）                                │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 PTES的意义

- **标准化流程**：确保测试的完整性和一致性
- **法律合规**：明确测试范围，避免法律风险
- **质量保证**：提供可重复、可验证的测试结果
- **沟通桥梁**：便于团队协作和客户沟通

## 二、前期交互阶段（Pre-Engagement Interactions）

### 2.1 阶段目标

前期交互阶段是渗透测试的起点，主要完成以下工作：
- 明确测试目标和范围
- 签署法律文件和授权协议
- 确定测试类型和方法
- 制定测试计划和时间表

### 2.2 关键文档

#### 2.2.1 授权书（Letter of Authorization）

授权书是渗透测试的法律基础，必须包含以下内容：

```
渗透测试授权书模板：

授权方：[公司名称]
被授权方：[测试团队/公司名称]
授权范围：
  - 目标系统：[具体IP/域名列表]
  - 测试类型：[黑盒/白盒/灰盒]
  - 测试时间：[起始日期] 至 [结束日期]
  - 禁止操作：[如DDoS、社工等]

授权人签字：_____________  日期：_____________
公司盖章：
```

#### 2.2.2 保密协议（NDA）

保密协议确保测试过程中获取的敏感信息不被泄露：

```markdown
保密协议要点：
1. 测试过程中获取的所有信息属于机密
2. 不得向第三方披露任何测试相关信息
3. 测试结束后销毁所有临时数据
4. 违约责任和赔偿条款
```

#### 2.2.3 测试范围定义

| 项目 | 说明 | 示例 |
|------|------|------|
| 目标系统 | 明确测试目标 | 192.168.1.0/24 |
| 排除系统 | 禁止测试的系统 | 生产数据库服务器 |
| 测试深度 | 测试的深入程度 | 应用层、网络层 |
| 时间窗口 | 允许测试的时间 | 工作日 22:00-06:00 |

### 2.3 测试类型选择

#### 黑盒测试（Black Box Testing）
- **特点**：测试者无任何目标系统信息
- **模拟**：外部攻击者的视角
- **优点**：真实模拟外部威胁
- **缺点**：耗时较长，可能遗漏内部漏洞

#### 白盒测试（White Box Testing）
- **特点**：测试者拥有完整系统信息
- **模拟**：内部人员或代码审计
- **优点**：测试全面，效率高
- **缺点**：成本较高

#### 灰盒测试（Gray Box Testing）
- **特点**：测试者拥有部分系统信息
- **模拟**：内部威胁或部分信息泄露场景
- **优点**：平衡效率和真实性
- **缺点**：需要合理的信息披露策略

### 2.4 实战注意事项

```bash
# 前期交互检查清单
□ 确认授权书已签署
□ 确认测试范围已明确界定
□ 确认禁止操作清单
□ 确认紧急联系人信息
□ 确认测试时间窗口
□ 确认IP白名单已配置
□ 准备测试环境和工具
□ 制定应急预案
```

## 三、信息收集阶段（Intelligence Gathering）

### 3.1 阶段目标

信息收集是渗透测试最重要的阶段之一，决定了后续测试的方向和效率。本阶段目标是尽可能多地收集目标系统的相关信息。

### 3.2 信息收集分类

```
信息收集
├── 被动信息收集（不直接接触目标）
│   ├── WHOIS信息查询
│   ├── 搜索引擎挖掘
│   ├── DNS历史记录
│   ├── 社交媒体信息
│   └── 公开泄露数据
│
└── 主动信息收集（直接接触目标）
    ├── 端口扫描
    ├── 服务识别
    ├── 系统指纹识别
    ├── 目录枚举
    └── 子域名探测
```

echo "[+] 漏洞扫描..."
nuclei -l subdomains.txt -t templates/ -o nuclei_results.txt

echo "[*] 信息收集完成"
```

## 四、威胁建模阶段（Threat Modeling）

### 4.1 阶段目标

威胁建模阶段基于信息收集的结果，分析目标系统可能存在的安全威胁，确定攻击路径和优先级。

### 4.2 威胁建模方法论

#### 4.2.1 STRIDE模型

| 威胁类型 | 说明 | 示例 |
|----------|------|------|
| **S**poofing（欺骗） | 冒充他人身份 | 钓鱼攻击、IP欺骗 |
| **T**ampering（篡改） | 恶意修改数据 | 数据注入、文件篡改 |
| **R**epudiation（抵赖） | 否认操作行为 | 日志删除、审计绕过 |
| **I**nformation Disclosure（信息泄露） | 未授权信息访问 | 敏感数据泄露 |
| **D**enial of Service（拒绝服务） | 影响服务可用性 | DDoS攻击 |
| **E**levation of Privilege（权限提升） | 获取更高权限 | 提权漏洞 |

#### 4.2.2 DREAD风险评估

```
DREAD评分 = (D + R + E + A + D) / 5

D (Damage) - 潜在损害程度 (1-10)
R (Reproducibility) - 复现难度 (1-10)
E (Exploitability) - 利用难度 (1-10)
A (Affected users) - 受影响用户数 (1-10)
D (Discoverability) - 发现难度 (1-10)

风险等级划分：
- 高风险：7-10分
- 中风险：4-6分
- 低风险：1-3分
```

### 4.3 攻击面分析

```markdown
攻击面分析框架：

1. 网络层攻击面
   - 开放端口和服务
   - 网络协议漏洞
   - 防火墙规则绕过

2. 应用层攻击面
   - Web应用漏洞
   - API接口安全
   - 认证授权机制

3. 主机层攻击面
   - 操作系统漏洞
   - 服务配置错误
   - 本地提权漏洞

4. 人员层攻击面
   - 社会工程学
   - 钓鱼攻击
   - 物理安全
```

### 4.4 攻击路径规划

```
示例攻击路径：

目标：获取数据库敏感数据

路径1（Web应用）：
信息收集 → SQL注入 → 数据库访问 → 数据导出

路径2（服务漏洞）：
端口扫描 → MySQL弱口令 → 数据库访问 → 数据导出

路径3（系统提权）：
Web漏洞 → Webshell → 系统提权 → 数据库配置获取 → 数据访问

优先级排序：
路径2 > 路径1 > 路径3（基于可行性和影响）
```

### 5.1 阶段目标

## 六、漏洞利用阶段（Exploitation）

### 6.1 阶段目标

漏洞利用阶段对已确认的漏洞进行实际攻击，获取目标系统的访问权限或敏感信息。

### 6.2 利用原则

```markdown
漏洞利用原则：

1. 授权范围内进行
2. 避免对系统造成不可逆损害
3. 保护已获取的敏感数据
4. 详细记录利用过程
5. 及时通知客户重大发现
```

### 6.3 常见漏洞利用技术

#### 6.3.1 Web漏洞利用

**SQL注入利用**

```bash

# 6. 获取Shell（如有权限）

## 七、后渗透阶段（Post Exploitation）

### 7.1 阶段目标

后渗透阶段在成功获取目标系统访问权限后，进行进一步的渗透和信息收集，包括权限提升、横向移动、信息收集等。

### 7.2 权限提升

#### 7.2.1 Linux提权

```bash

### 8.3 漏洞报告模板

```markdown
### 漏洞名称：SQL注入漏洞

**漏洞等级**：高危

**漏洞位置**：http://example.com/page?id=1

**漏洞描述**：
在应用程序的查询参数处存在SQL注入漏洞，攻击者可以通过构造恶意SQL语句获取数据库敏感信息，包括用户凭据、业务数据等。

**复现步骤**：
1. 访问目标URL：http://example.com/page?id=1
2. 添加单引号测试：http://example.com/page?id=1'
3. 观察返回错误信息，确认存在注入点
4. 使用sqlmap获取数据库信息：
   ```
   sqlmap -u "http://example.com/page?id=1" --dbs
   ```

**PoC代码**：
```python
import requests

url = "http://example.com/page"
payload = "1' UNION SELECT 1,2,3--+"

response = requests.get(url, params={"id": payload})
print(response.text)
```

**风险分析**：
- 攻击者可获取数据库全部数据
- 可能获取管理员账号密码
- 可能写入WebShell获取服务器权限
- CVSS评分：9.8（Critical）

**修复建议**：
1. 使用参数化查询，避免SQL拼接
2. 对用户输入进行严格过滤
3. 使用最小权限数据库账户
4. 部署WAF进行防护

**修复代码示例**：
```php
// 不安全的代码
$sql = "SELECT * FROM users WHERE id = " . $_GET['id'];

// 安全的代码（使用PDO参数化查询）
$stmt = $pdo->prepare("SELECT * FROM users WHERE id = :id");
$stmt->bindParam(':id', $_GET['id'], PDO::PARAM_INT);
$stmt->execute();
```

**参考链接**：
- OWASP SQL Injection: https://owasp.org/www-community/attacks/SQL_Injection
- CWE-89: https://cwe.mitre.org/data/definitions/89.html
```

### 8.4 风险评级标准

| 等级 | CVSS评分 | 描述 | 处理时限 |
|------|----------|------|----------|
| 严重 | 9.0-10.0 | 可直接获取系统控制权 | 24小时内 |
| 高危 | 7.0-8.9 | 可获取敏感数据或权限 | 7天内 |
| 中危 | 4.0-6.9 | 可能造成一定影响 | 30天内 |
| 低危 | 0.1-3.9 | 影响较小 | 90天内 |
| 信息 | 0.0 | 无安全风险 | 可选修复 |
