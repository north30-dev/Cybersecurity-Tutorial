# 容器安全

## 一、Docker安全概述

### 1.1 容器安全的重要性

容器技术（以Docker为代表）已成为现代应用部署的标准方式，但其安全性往往被忽视。容器并非完全隔离的"沙箱"，而是共享宿主机内核的轻量级进程隔离技术。理解容器安全的本质是构建安全云原生环境的基础。

**容器安全的三大支柱：**

```
┌─────────────────────────────────────────────────────────┐
│                    容器安全体系                          │
├─────────────────┬─────────────────┬─────────────────────┤
│    镜像安全      │    运行时安全    │    宿主机安全        │
├─────────────────┼─────────────────┼─────────────────────┤
│ • 镜像扫描       │ • 资源限制      │ • 内核安全          │
│ • 签名验证       │ • 能力控制      │ • 用户命名空间      │
│ • 最小化镜像     │ • 网络隔离      │ • 安全模块          │
│ • 可信来源       │ • 日志监控      │ • 及时更新          │
└─────────────────┴─────────────────┴─────────────────────┘
```

### 1.2 Docker安全架构

Docker利用Linux内核的多项安全特性实现隔离：

| 安全特性 | 作用 | Docker默认配置 |
|---------|------|---------------|
| Namespaces | 资源隔离（PID、NET、MNT等） | 启用 |
| Cgroups | 资源限制（CPU、内存、IO） | 部分启用 |
| Capabilities | 权限细分控制 | 限制部分能力 |
| Seccomp | 系统调用过滤 | 启用默认配置 |
| AppArmor/SELinux | 强制访问控制 | 可选启用 |

### 1.3 容器与虚拟机安全对比

```
┌──────────────────────────────────────────────────────────┐
│                    虚拟机架构                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │   App A     │  │   App B     │  │   App C     │       │
│  │  Bins/Libs  │  │  Bins/Libs  │  │  Bins/Libs  │       │
│  │ Guest OS    │  │ Guest OS    │  │ Guest OS    │       │
│  └─────────────┘  └─────────────┘  └─────────────┘       │
│  ┌──────────────────────────────────────────────┐        │
│  │              Hypervisor                       │       │
│  └──────────────────────────────────────────────┘        │
│  ┌──────────────────────────────────────────────┐        │
│  │              Host OS                          │       │
│  └──────────────────────────────────────────────┘        │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                    容器架构                               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │   App A     │  │   App B     │  │   App C     │       │
│  │  Bins/Libs  │  │  Bins/Libs  │  │  Bins/Libs  │       │
│  └─────────────┴──┴─────────────┴──┴─────────────┘       │
│  ┌──────────────────────────────────────────────┐        │
│  │         Container Runtime (Docker)            │       │
│  └──────────────────────────────────────────────┘        │
│  ┌──────────────────────────────────────────────┐        │
│  │              Host OS (共享内核)               │       │
│  └──────────────────────────────────────────────┘        │
└──────────────────────────────────────────────────────────┘
```

**关键安全差异：**
- 虚拟机：独立内核，强隔离，攻击面小
- 容器：共享内核，弱隔离，需额外安全措施

## 二、镜像安全

### 2.1 镜像安全扫描

#### 2.1.1 漏洞扫描工具

**Trivy - 推荐的开源扫描工具：**

```bash
# 安装Trivy
curl -sf https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin

# 扫描镜像
trivy image nginx:latest

# 扫描并输出JSON格式
trivy image --format json --output results.json nginx:latest

# 只显示HIGH和CRITICAL级别漏洞
trivy image --severity HIGH,CRITICAL nginx:latest

# 扫描镜像并退出码非零（用于CI/CD）
trivy image --exit-code 1 --severity HIGH,CRITICAL nginx:latest
```

#### 2.1.2 CI/CD集成扫描

**GitLab CI配置示例：**

```yaml
# .gitlab-ci.yml
container_scanning:
  stage: test
  image: 
    name: aquasec/trivy:latest
    entrypoint: [""]
  variables:
    TRIVY_NO_PROGRESS: "true"
    TRIVY_SEVERITY: "HIGH,CRITICAL"
  script:
    - trivy image --exit-code 1 --severity $TRIVY_SEVERITY $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
  allow_failure: false
```

**GitHub Actions配置示例：**

```yaml
# .github/workflows/container-scan.yml
name: Container Security Scan
on: [push, pull_request]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'table'
          exit-code: '1'
          severity: 'CRITICAL,HIGH'
```

### 2.2 镜像签名验证

#### 2.2.1 Docker Content Trust (DCT)

```bash
# 启用Docker Content Trust
export DOCKER_CONTENT_TRUST=1

# 签名推送镜像
docker push myregistry/myimage:latest
# 首次使用会生成签名密钥

# 拉取时自动验证签名
docker pull myregistry/myimage:latest

# 查看签名信息
docker trust inspect --pretty myregistry/myimage:latest
```

#### 2.2.2 Cosign (推荐方案)

```bash
# 安装cosign
go install github.com/sigstore/cosign/v2/cmd/cosign@latest

# 生成密钥对
cosign generate-key-pair

# 签名镜像
cosign sign --key cosign.key myregistry/myimage@sha256:xxx

# 验证签名
cosign verify --key cosign.pub myregistry/myimage@sha256:xxx

# 在Kubernetes中验证（使用Kyverno策略）
```

### 2.3 最小化镜像

#### 2.3.1 基础镜像选择策略

| 基础镜像 | 大小 | 安全性 | 适用场景 |
|---------|------|--------|---------|
| ubuntu:latest | ~72MB | 中等 | 开发测试 |
| debian:slim | ~80MB | 较好 | 生产环境 |
| alpine:latest | ~5MB | 好（注意glibc兼容） | 追求极致小 |
| distroless/base | ~20MB | 最佳 | 生产环境首选 |
| scratch | 0MB | 最佳 | 静态编译程序 |

#### 2.3.2 多阶段构建示例

```dockerfile
# 构建阶段
FROM golang:1.21-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main .

# 运行阶段 - 使用distroless
FROM gcr.io/distroless/static:nonroot

COPY --from=builder /app/main /
USER nonroot:nonroot

ENTRYPOINT ["/main"]
```

#### 2.3.3 镜像瘦身最佳实践

```dockerfile
# 不推荐：包含构建工具和缓存
FROM node:18
WORKDIR /app
COPY . .
RUN npm install
CMD ["npm", "start"]

# 推荐：多阶段构建，仅包含运行时
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
RUN npm cache clean --force && \
    rm -rf /tmp/* /var/cache/apk/*
USER node
CMD ["node", "server.js"]
```

## 三、容器配置安全

### 3.1 特权容器风险

#### 3.1.1 特权模式危险

```bash
# 危险：特权容器
docker run --privileged nginx

# 特权容器可以：
# - 访问所有设备
# - 修改内核模块
# - 绕过所有安全机制
# - 可能实现容器逃逸
```

**特权容器攻击示例：**

```bash
# 在特权容器内挂载宿主机根文件系统
docker run --privileged -it alpine sh
# 容器内执行：
mount /dev/sda1 /mnt
chroot /mnt
# 此时已获得宿主机root权限
```

#### 3.1.2 安全替代方案

```yaml
# 仅添加需要的能力，而非全部
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  containers:
  - name: app
    image: nginx
    securityContext:
      privileged: false
      capabilities:
        add: ["NET_BIND_SERVICE"]  # 仅添加需要的能力
        drop: ["ALL"]              # 先移除所有能力
```

### 3.2 能力限制（Capabilities）

#### 3.2.1 Linux Capabilities概述

```
┌─────────────────────────────────────────────────────────┐
│              Linux Capabilities 分类                     │
├─────────────────────────────────────────────────────────┤
│  文件系统相关：                                          │
│  • CAP_CHOWN - 修改文件所有者                           │
│  • CAP_DAC_OVERRIDE - 绕过文件权限检查                   │
│  • CAP_FOWNER - 绕过文件所有者检查                       │
├─────────────────────────────────────────────────────────┤
│  网络相关：                                              │
│  • CAP_NET_BIND_SERVICE - 绑定特权端口                   │
│  • CAP_NET_RAW - 使用原始套接字                         │
│  • CAP_NET_ADMIN - 网络配置管理                         │
├─────────────────────────────────────────────────────────┤
│  系统相关：                                              │
│  • CAP_SYS_ADMIN - 强大的系统管理能力                   │
│  • CAP_SYS_TIME - 修改系统时间                          │
│  • CAP_SYS_MODULE - 加载内核模块                        │
└─────────────────────────────────────────────────────────┘
```

#### 3.2.2 能力配置实践

```yaml
# 安全的容器能力配置
apiVersion: v1
kind: Pod
metadata:
  name: web-server
spec:
  containers:
  - name: nginx
    image: nginx:latest
    securityContext:
      runAsNonRoot: true
      runAsUser: 101  # nginx用户
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
          - ALL       # 移除所有能力
        add:
          - NET_BIND_SERVICE  # 仅添加绑定80/443端口的能力
          - CHOWN             # 日志文件权限修改
```

### 3.3 资源限制

#### 3.3.1 资源限制配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-limited-pod
spec:
  containers:
  - name: app
    image: myapp:latest
    resources:
      requests:      # 保证的最小资源
        memory: "256Mi"
        cpu: "250m"
      limits:        # 最大资源限制
        memory: "512Mi"
        cpu: "500m"
    # 防止资源耗尽攻击
```

#### 3.3.2 Docker资源限制

```bash
# 内存限制
docker run -d \
  --memory="512m" \
  --memory-swap="512m" \
  --memory-reservation="256m" \
  --oom-kill-disable=false \
  myapp:latest

# CPU限制
docker run -d \
  --cpus="1.5" \
  --cpu-shares=512 \
  --cpuset-cpus="0-2" \
  myapp:latest

# IO限制
docker run -d \
  --device-read-bps=/dev/sda:10mb \
  --device-write-bps=/dev/sda:10mb \
  --device-read-iops=/dev/sda:1000 \
  myapp:latest
```

## 四、容器逃逸漏洞

### 4.1 重大CVE案例分析

#### 4.1.1 CVE-2019-5736 (runc逃逸)

**漏洞描述：** 攻击者可通过特制容器镜像覆盖宿主机的runc二进制文件，获得宿主机root权限。

**影响版本：** runc < 1.0.0-rc6

**攻击原理：**

```
┌─────────────────────────────────────────────────────────┐
│           CVE-2019-5736 攻击流程                         │
├─────────────────────────────────────────────────────────┤
│  1. 恶意镜像包含 /bin/sh 指向 /proc/self/exe            │
│  2. 容器启动时，runc打开容器内的/bin/sh                 │
│  3. 由于符号链接，实际打开的是宿主机runc                 │
│  4. 攻击者通过文件描述符覆盖runc二进制                   │
│  5. 下次执行docker exec时运行恶意代码                    │
└─────────────────────────────────────────────────────────┘
```

**修复措施：**

```bash
# 升级runc到安全版本
apt-get update && apt-get install -y runc

# 或升级Docker
curl -fsSL https://get.docker.com | sh
```

#### 4.1.2 CVE-2020-8558 (Kubernetes kube-proxy)

**漏洞描述：** kube-proxy的IPv6实现缺陷允许从集群内部访问节点上的localhost服务。

**影响版本：** Kubernetes v1.18.0-v1.18.5, v1.17.0-v1.17.8, v1.16.0-v1.16.12

**缓解措施：**

```bash
# 确保节点防火墙规则正确
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -j DROP
```

#### 4.1.3 CVE-2022-0847 (Dirty Pipe)

**漏洞描述：** Linux内核漏洞，允许非特权用户覆盖任意只读文件。

**影响版本：** Linux内核 5.8 - 5.16

**容器环境风险：** 容器内进程可利用此漏洞修改宿主机关键文件。

**检测与修复：**

```bash
# 检查内核版本
uname -r

# 升级内核到修复版本
# Ubuntu/Debian
apt-get update && apt-get install linux-image-generic

# RHEL/CentOS
yum update kernel
```

### 4.2 逃逸防护措施

```yaml
# 综合安全配置
apiVersion: v1
kind: Pod
metadata:
  name: escape-resistant-pod
  annotations:
    container.apparmor.security.beta.kubernetes.io/app: runtime/default
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myapp:latest
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
```

## 五、安全配置最佳实践

### 5.1 Docker Daemon安全配置

```json
// /etc/docker/daemon.json
{
  "icc": false,                    // 禁用容器间通信
  "live-restore": true,            // 守护进程重启时保持容器运行
  "userns-remap": "default",       // 启用用户命名空间
  "no-new-privileges": true,       // 禁止权限提升
  "default-ulimits": {
    "nofile": {
      "Name": "nofile",
      "Hard": 65536,
      "Soft": 65536
    }
  },
  "storage-driver": "overlay2",
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "metrics-addr": "127.0.0.1:9323"  // 仅本地访问
}
```

### 5.2 Dockerfile安全最佳实践

```dockerfile
# 1. 使用特定版本的基础镜像
FROM alpine:3.18.0

# 2. 创建非root用户
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# 3. 设置工作目录
WORKDIR /app

# 4. 复制必要文件并设置权限
COPY --chown=appuser:appgroup app .

# 5. 安装依赖后清理缓存
RUN apk add --no-cache openssl && \
    rm -rf /var/cache/apk/*

# 6. 设置安全环境变量
ENV PATH=/app/bin:$PATH \
    PYTHONUNBUFFERED=1

# 7. 切换到非root用户
USER appuser

# 8. 只暴露必要的端口
EXPOSE 8080

# 9. 健康检查
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -q --spider http://localhost:8080/health || exit 1

# 10. 使用exec形式启动
ENTRYPOINT ["./app"]
CMD ["--config", "/app/config.yaml"]
```

### 5.3 docker-compose安全配置

```yaml
version: '3.8'

services:
  app:
    image: myapp:1.0.0
    container_name: myapp
    
    # 安全配置
    security_opt:
      - no-new-privileges:true
      - seccomp:unconfined  # 仅在必要时使用
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    
    # 用户配置
    user: "1000:1000"
    
    # 资源限制
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
    
    # 只读文件系统
    read_only: true
    tmpfs:
      - /tmp
      - /var/run
    
    # 网络配置
    networks:
      - appnet
    ports:
      - "8080:8080"
    
    # 健康检查
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 3s
      retries: 3

networks:
  appnet:
    driver: bridge
    internal: false
```

## 六、运行时安全监控

### 6.1 容器运行时安全工具

#### 6.1.1 Falco - 运行时安全监控

**安装Falco：**

```bash
# Debian/Ubuntu
curl -s https://falco.org/repo/falcosecurity-packages.asc | apt-key add -
echo "deb https://download.falco.org/packages/deb stable main" | tee -a /etc/apt/sources.list.d/falcosecurity.list
apt-get update && apt-get install -y falco

# 启动Falco
systemctl enable falco
systemctl start falco
```

**Falco规则示例：**

```yaml
# /etc/falco/falco_rules.local.yaml

# 检测特权容器启动
- rule: Launch Privileged Container
  desc: Detect privileged container start
  condition: container_started and container.privileged = true
  output: Privileged container started (user=%user.name container=%container.id image=%container.image.repository)
  priority: WARNING
  tags: [container, privilege]

# 检测容器内执行shell
- rule: Shell Spawned in Container
  desc: Detect shell execution in container
  condition: spawned_process and container and proc.name in (bash, sh, zsh)
  output: Shell spawned in container (user=%user.name container=%container.id shell=%proc.name)
  priority: NOTICE
  tags: [container, shell]

# 检测敏感文件访问
- rule: Read Sensitive File in Container
  desc: Detect reading of sensitive files
  condition: open_read and container and fd.name in (/etc/shadow, /etc/passwd, /root/.ssh/id_rsa)
  output: Sensitive file read in container (user=%user.name container=%container.id file=%fd.name)
  priority: WARNING
  tags: [container, filesystem]
```

## 七、安全检查清单

### 7.1 镜像安全检查

- [ ] 使用官方或可信基础镜像
- [ ] 镜像已进行漏洞扫描并修复高危漏洞
- [ ] 镜像已签名并验证
- [ ] 使用最小化镜像（alpine/distroless）
- [ ] 镜像不包含敏感信息（密钥、密码）
- [ ] 使用多阶段构建减少攻击面

### 7.2 容器配置检查

- [ ] 不使用特权模式（--privileged）
- [ ] 以非root用户运行
- [ ] 限制容器能力（capabilities）
- [ ] 设置资源限制（CPU、内存）
- [ ] 使用只读根文件系统
- [ ] 禁止权限提升（no-new-privileges）
- [ ] 挂载卷使用最小权限

### 7.3 运行时安全检查

- [ ] 启用运行时安全监控（Falco）
- [ ] 配置审计日志
- [ ] 实施网络隔离
- [ ] 定期安全扫描
- [ ] 制定事件响应计划
- [ ] 保持运行时组件更新
